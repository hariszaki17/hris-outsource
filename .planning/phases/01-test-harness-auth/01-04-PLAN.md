<interfaces>
Generated hooks already exist (verified) in `@swp/api-client` (barrel `@swp/api-client/e1`):
```ts
useAuthLogin()          // POST /auth/login   ; mutate({ data: LoginRequest })  → LoginResponse
useAuthForgotPassword() // POST /auth/forgot-password ; mutate({ data: { email } }) → 202 { message }
useAuthResetPassword()  // POST /auth/reset-password  ; mutate({ data: { reset_token, new_password } }) → 204
useAuthLogout()         // POST /auth/logout  ; mutate() → 204
useAuthMe()             // GET /auth/me       → MeResponse
```
LoginRequest = { email: string; password: string; stay_signed_in?: boolean }.
LoginResponse = { access_token, refresh_token, token_type, expires_in, user: MeResponse }.
MeResponse = { id, email, role, status, employee_id, full_name, last_login_at?, scope?: { type: 'global'|'company'|'self', company_id?: string|null } }.

Errors: the mutator throws `ApiError` (frontend/packages/api-client/src/errors.ts) on non-2xx; it carries the parsed envelope. The error envelope code is at `error.code` (e.g. INVALID_CREDENTIALS, ACCOUNT_DISABLED, ACCOUNT_LOCKED, RESET_TOKEN_EXPIRED, WEAK_PASSWORD, UNAUTHENTICATED). Check `ApiError` for the exact field exposing the code/status.

Existing FE auth store (frontend/apps/web/src/lib/auth.ts):
```ts
interface SessionUser { name: string; role: Role; permissions: readonly Permission[]; initials: string; companyName?: string }
auth.login(token, user); auth.clear(); auth.getToken(); auth.subscribe(fn);
installAuth() // calls configureApiClient({ baseUrl: VITE_API_BASE_URL ?? '/api/v1', getToken, onUnauthenticated })
permissionsForRole(role) // from @swp/shared — derive permissions
```
Current login-screen.tsx STUBS `auth.login('dev-token', {...hardcoded Sari Hadi...})` — REMOVE this.
Login error UI is driven by a `?error` search param with values 'invalid' | 'locked' | 'disabled' (Banners already built). Map: INVALID_CREDENTIALS→'invalid', ACCOUNT_LOCKED/429→'locked', ACCOUNT_DISABLED→'disabled'.

Mutator (frontend/packages/api-client/src/mutator.ts) currently calls `fetch(fullUrl, { ...options, headers })` WITHOUT `credentials`. For the cookie refresh transport cross-origin (FE :4173 → BE :8081) it MUST send `credentials: 'include'`. The BE sets CORS allow-origin :4173 and cookie SameSite=Lax, Secure=false in the test env.

Routes (frontend/apps/web/src/app/router.tsx): `/login` (search may include error), `/forgot-password`, `/reset-password`, index `/` (the dashboard target after login), authed routes redirect to `/login` when unauthenticated. `/reset-password` needs a typed `token` search param (add validateSearch).
