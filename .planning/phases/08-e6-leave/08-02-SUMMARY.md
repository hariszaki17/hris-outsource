# Phase 8 Plan 02: E6 Leave Services + Handlers + INV-3 Loop-Closer Summary

Implemented the E6 **service + handler layer** for the 10 FE-used leave endpoints,
the **two-level approval state machine** (PENDING_L1 → PENDING_HR → APPROVED; reject
→ REJECTED), the **soft-reservation quota model** (QUOTA_EXCEEDED + BALANCE_RECHECK_FAILED
guards, adjust, bulk-grant partial-success/pro-rate), the **calendar**, and — the
gnarly part — the cross-epic **INV-3 loop-closer**: on `:approve-final` / `:approve-override`
the service cancels overlapping `schedule_entries` (DB `CANCELLED_BY_LEAVE`, surfaced
to the contract as `new_status:'LEAVE'`) AND inserts the real `approved_leave_days`
rows in the SAME tx, fulfilling the production-source hand-off Phase-6 documented.
`make gen` + `go build ./...` + `go vet ./...` clean; the service unit tests + the
existing Phase-7 suite pass; no regressions.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] INV-3 port type identity across packages**
- **Found during:** Task 1
- **Issue:** The plan suggested the scheduling repo return its own impact type and the
  leave service import a separately-declared `SchedulePort`. Go's structural interface
  satisfaction requires *identical* named return types — a scheduling-package-local
  `[]leaveScheduleImpact` cannot satisfy a `service/leave` interface returning
  `[]svc.ScheduleImpact`. A reflection-free adapter was also impossible for the same reason.
- **Fix:** Defined `SchedulePort` (and `ScheduleImpact`) in `service/leave`; the
  scheduling repo imports `service/leave` and returns `[]leavesvc.ScheduleImpact`
  directly. No import cycle (service/leave never imports repository/scheduling). Removed
  the abandoned adapter file.
- **Files modified:** repository/scheduling/schedule_repo.go, service/leave/ports.go
- **Commit:** `32b6866`
