<assertions_required>
From docs/api/E5-attendance/openapi.yaml — assert exact status + code:
  - GET /attendance → 200, envelope {data:[Attendance], next_cursor, has_more}; leader-scope filters
    to own company; leader passing company_id=CMP-0022 → 403 OUT_OF_SCOPE.
  - GET /attendance/{id} → 200 {data: Attendance}; cross-scope → 404 (hides existence).
  - POST /attendance/{id}:verify → 200 {data}; leader verifying own ESCALATED record → 403 VERIFY_OWN_RECORD;
    leader verifying CMP-0022 record → 403 OUT_OF_SCOPE; already-VERIFIED → 409 (code CONFLICT, fields.verification_status).
  - POST /attendance/{id}:reject → 200 {data}; missing/short reason → 400 INVALID_REQUEST (fields.reason);
    already-decided → 409.
  - POST /attendance:bulk-verify → 200 with {succeeded:[ids], failed:[{id, error{code,message}}]} where a
    mix of a valid PENDING id + the leader's own ESCALATED id yields one succeeded + one failed
    (VERIFY_OWN_RECORD); all-fail set → 422.
  - POST /attendance:bulk-reject → analogous 200 partial + 422 all-fail.
  - Idempotency replay: the idempotency middleware is router-level; assert it is mounted (a second
    identical request with the same Idempotency-Key returns the SAME body/status — drive via the harness
    that wires d.Idempotency.Handler, OR assert the route registration includes the middleware). If the
    fake harness cannot exercise the real Postgres-backed store, assert IDEMPOTENCY_KEY_REUSED 409
    behavior at the middleware boundary with a stub store; document the seam in the SUMMARY.
  - GET /corrections → 200 {data:[Correction], next_cursor, has_more}; leader scope by company.
  - GET /corrections/{id} → 200 {data: Correction with diff[]}; assert diff rows for a CHECK_OUT correction.
  - POST /corrections/{id}:approve → 200 {data: Correction(status APPLIED), attendance: Attendance(CORRECTED flag)};
    non-PENDING → 409; non-HR caller outside 7-day window → 422 OUTSIDE_CORRECTION_WINDOW
    (fields.attendance_date, window_days="7") via the exported CheckCorrectionWindow seam or an approve on a
    stale-dated PENDING correction by a leader principal.
  - POST /corrections/{id}:reject → 200 {data: status REJECTED}; missing reason → 400; non-PENDING → 409.
  - CORRECTION_ALREADY_PENDING (409, fields.pending_correction_id): assert via the service guard / unique-index
    path using two PENDING corrections on one attendance in the fake repo (the partial-unique-index backstop
    is integration-only; here assert the service pre-check returns the code).
</assertions_required>
