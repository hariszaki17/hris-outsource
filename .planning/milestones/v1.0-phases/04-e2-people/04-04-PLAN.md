Service/handler/repo skeleton: identical to 04-02/04-03 (ChangeRequestService as a separate
struct in package people; ChangeRequestHandler in package people).

Error codes:
- approve/reject on a request whose status != 'pending' → `apperr.Conflict("CONFLICT")` → 409.
- reject with missing/short reason (<3 chars) → `apperr.Invalid({"reason":"..."})` → 400.
- request not found → `apperr.NotFound()` → 404.

OpenAPI ChangeRequest schema (openapi.yaml 2893-2926), snake_case:
id, employee_id, status(PENDING|APPROVED|REJECTED), submitted_at, resolved_at(nullable),
resolved_by(nullable), rejection_reason(nullable), note(nullable),
changes{phone?,address?,bank_account?}, request_type(PHONE|ADDRESS|BANK_ACCOUNT|MULTIPLE).
ChangeRequestDetail (2943): id, employee{id,full_name,nip}, status, submitted_at,
resolved_at, resolved_by, rejection_reason, note, diff{<field>:{old,new}}.
:reject body (770-782): {reason} required, minLength 3, maxLength 500.
:approve has no body (idempotency-key optional).

GET /change-requests filters (openapi.yaml 628-649): status, employee_id, request_type, q.
List + detail both ordered by submitted_at desc (cursor on submitted_at,id from 04-01 query).

Queries available (04-01 db/queries/people/change_requests.sql):
ListChangeRequests, GetChangeRequestByID, CreateChangeRequest, ResolveChangeRequest.
For approve-applies-change you also need to UPDATE the employee — reuse the employees
repo's UpdateEmployee (add the needed read/write methods to the ChangeRequestRepository
interface OR inject the employee repo into the change-request service).
